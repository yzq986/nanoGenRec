# CRAB: Codebook Rebalancing for Bias Mitigation in Generative Recommendation

- **Arxiv**: 2604.05113
- **Authors**: Ziheng Chen, Zezhong Fan, Luyi Ma, Jin Huang, Lalitesh Morishetti, Kaushiki Nag, Sushant Kumar, Kannan Achan (Walmart Global Tech)
- **Date**: 2026-04
- **Read date**: 2026-04-22

## Core Contribution
- First systematic investigation showing that **imbalanced codebooks create over-popular tokens** that amplify popularity bias in generative recommendation—top 5% tokens cause 1.8× more exposure bias vs SASRec (GU gap of 0.42 on MOR)
- Proposes CRAB, a **post-hoc two-stage debiasing** method: (1) split over-popular tokens via regularized K-means on child-token redistribution with a popularity-balancing penalty; (2) hierarchical semantic alignment regularizer (L_T) to transfer knowledge to new/unpopular tokens
- Achieves **16.5% reduction in MGU@10** (popularity bias) while maintaining or slightly improving recommendation accuracy (HR@10, NDCG@10), with only 0.28h training time vs 2.75-3.11h for competing debiasing methods

## Method
CRAB operates as a post-hoc correction on a trained generative recommender with an existing RQ codebook. **Stage 1 (Codebook Rebalancing)**: Identify over-popular tokens (top 10% by cumulative item interaction frequency) at each codebook level. Split each over-popular token into M≤3 new tokens by solving a regularized K-means problem that redistributes its child tokens at the next level, subject to a balancing loss (L_bal) that penalizes popularity variance among the new tokens. The hard constraint preserves semantic integrity: all items sharing the same (l+1)-th level token stay together. **Stage 2 (Hierarchical Semantic Alignment)**: Fine-tune the LLM with a combined loss L = L_Rec + γ·L_T (γ=0.2), where L_T is a tree-structure-aware regularizer encouraging sibling tokens (same parent) to have similar LLM embeddings. This enhances under-represented token representations via semantic supervision from related siblings. Only LoRA (r=8, α=16) on attention layers + embedding layers are updated, making training efficient (~0.28h on 4×A100).

## Key Results

| Metric | MOR (baseline) | Tiger | RW | RR | D2LR | **CRAB** |
|--------|---------------|-------|-----|-----|------|----------|
| HR@10 ↑ | 0.152 | 0.132 | 0.126 | 0.141 | 0.147 | **0.152** |
| NDCG@10 ↑ | 0.116 | 0.090 | 0.070 | 0.107 | 0.113 | **0.117** |
| DGU@10 ↓ | 0.418 | 0.423 | 0.367 | 0.406 | 0.410 | **0.356** |
| MGU@10 ↓ | 0.109 | 0.112 | 0.105 | 0.109 | 0.106 | **0.091** |
| Time (h) ↓ | - | - | 3.11 | 0.21 | 2.75 | **0.28** |

Key ablation findings:
- **Splitting ratio**: 10% is optimal; <10% under-corrects, >15% disrupts codebook semantic integrity
- **Splitting position**: Intermediate level (L1/B) is most effective for debiasing, consistent with the "Hourglass" phenomenon where mid-level tokens concentrate excessive semantic information; splitting the last level hurts performance (already fine-grained)
- **γ (regularizer weight)**: γ≤0.2 stable; γ>0.2 causes sharp NDCG drop
- **LoRA is necessary**: Removing LoRA causes noticeable drops in both NDCG and HR

## Relevance to nanoGenRec
The nanoGenRec system uses a 3-level hierarchical SID (L0→L1→L2) with RQ/MLP-FSQ tokenizer, NTP transformer, and beam search decoding. CRAB is directly applicable:

1. **Token popularity imbalance exists in nanoGenRec**: The current tokenizer has KMeans Gini=0.31 for the first two levels (noted in ideas/tokenizer.md). CRAB's insight that popular tokens amplify bias during NTP autoregressive training applies directly—beam search will over-generate items under popular L0/L1 prefixes.

2. **Codebook rebalancing for L0/L1 KMeans layers**: CRAB's regularized K-means splitting can be applied post-hoc to the first two KMeans layers (1024 clusters each). Split the top 10% over-popular clusters into M≤3 sub-clusters by redistributing child tokens from the next level. This is a lightweight operation on the existing codebook (no re-tokenization of embeddings needed, just reassignment).

3. **Hierarchical semantic regularizer during NTP fine-tune**: L_T loss (Eq. 9) can be added to the existing NTP training loss. Since nanoGenRec already uses LoRA for NTP, the training setup matches CRAB's implementation exactly. γ=0.2 is a safe default.

4. **Expected impact**: CRAB maintains HR@10 while reducing popularity bias 16.5% (MGU). For nanoGenRec's R@500=61.2%, the main expected gain is **improved long-tail recall** (more diverse recommendations), not necessarily higher aggregate R@500. However, if the current beam search over-concentrates on popular prefixes, rebalancing could unlock recall on tail items that are currently unreachable, potentially improving R@500 by 1-3%.

5. **Splitting the intermediate level (L1) is most impactful** per CRAB's ablation, which aligns with the "Hourglass" observation. For nanoGenRec's 3-level structure, L1 (the middle KMeans layer) should be the priority target.

6. **Low implementation cost**: Post-hoc method, ~0.28h fine-tuning on 4×A100. No changes to the tokenizer training pipeline or the NTP architecture—just codebook surgery + LoRA fine-tuning with an added regularizer.
