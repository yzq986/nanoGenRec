## What questions does the SP-DPO leave?

The chosen of SP-DPO is the ground truth, and the rejected is the generation of the model itself.
The evaluation criterion is **purely structural** - prefix n-gram matching degree.

Problem: **It only knows "whether the prediction is correct", but not "whether the recommendation is good or not"**.

For example: The model generates candidate B. Although it is not ground truth, the user may also like it (it is just not exposed).
SP-DPO suppresses B as a negative sample - this is unjust.
On the other hand, the ground truth may be low-quality content that users randomly click on, but SP-DPO treats it as a positive sample and pushes it up.

> SP-DPO aligns on "prediction accuracy", not "recommendation quality".

## RF-DPO: Introducing real user feedback

**Source**: Align³GR (Kuaishou, AAAI 2026 Oral, arxiv 2511.11255)

**Core Improvement**: Replaced SP-DPO's structural signals with **Real User Behavior**.

```
SP-DPO: chosen = ground truth, rejected = generated by the model (scored by prefix matching)
RF-DPO: chosen = what users like, rejected = what users don’t like (real feedback)
```

User feedback is divided into three levels:

| Feedback | Signal | Example |
|------|------|------|
| Liked | Explicitly positive | Like, purchase, collect |
| Neutral | Implicit negative | Exposure without clicks |
| Disliked | Explicit negative | Clicked "Not interested" |

RF-DPO also progresses in two stages:

```
Easy stage: chosen = liked, rejected = disliked (easy to distinguish)
Hard stage: chosen = liked, rejected = neutral (Subtle distinction: Why was it exposed but not clicked?)
```

### Key differences from SP-DPO

| | SP-DPO | RF-DPO |
|--|--------|--------|
| Signal Source | Model Self-Game | User’s Real Behavior |
| Judgment criteria | Is the prediction accurate | Does the user like it or not |
| Align Goals | Generate Accuracy | **Business Goals** (CTR/Conversions/Retention) |
| Negative sample quality | Good items may be accidentally damaged | Defined by user behavior, more reliable |

### Ablation results (Align³GR, Instruments dataset)

| Method | Recall@10 | vs SP-DPO |
|------|-----------|-----------|
| Progressive SP-DPO | 0.1396 | — |
| + RF-DPO (no progression) | 0.1414 | +1.3% |
| + RF-DPO (with gradient) | **0.1442** | **+3.3%** |

### Limitations of RF-DPO

RF-DPO or DPO - it only looks at pairwise preferences (chosen vs rejected),
Each sample has only 1 chosen and 20 rejected.
If the model generates 500 candidates, the DPO only uses 21 of them, and all information about the remaining 479 is lost.

## GRPO: from pairwise to group-wise

**Source**: OneMall (arxiv 2601.21770), OneRec (arxiv 2506.13695v4)

### Limitations of DPO

Only one pair (chosen vs rejected) is compared each time, which results in low information efficiency.

### GRPO approach: Calculate relative advantages for the entire set of candidates

```
DPO: 1 chosen vs N rejected → pairwise comparison
GRPO: G candidates, each with reward score → group-wise relative ranking
```

### GRPO Process

```
Step 1: For each user, model beam search generates G candidates (G=512)
Step 2: reward model scores each candidate → r_1, r_2, ..., r_G
Step 3: Normalize within the group and calculate advantage:
        A_i = (r_i - mean(r)) / std(r)
Step 4: Optimize with clipped surrogate loss (same clip mechanism as PPO)
```

### The core difference with DPO

| | DPO | GRPO |
|--|-----|------|
| Comparison method | 1 chosen vs N rejected (dichotomy) | G candidate continuous ranking (relative advantage) |
| Information utilization | Only use the best and worst | **All candidates generate Gradient** |
| Reward Source | Implicit (partially Good vs. implicit) | **Explicit reward model scoring** |
| Optimization goal | Widen the probability gap between chosen/rejected | Adjust the probabilities of all candidates by advantage weighting |

### Specific examples

The model generated 5 candidates, reward model scored:

| Candidate | Reward | Advantage (after normalization) | What GRPO does |
|------|--------|---------------------|------------|
| A | 0.9 | +1.5 | Largely push High probability |
| B | 0.7 | +0.5 | Moderately recommend High |
| C | 0.5 | 0.0 | Barely moving |
| D | 0.3 | -0.5 | Moderate pressure Low |
| E | 0.1 | -1.5 | Significantly suppress Low |

The DPO will only see A (chosen) vs E (rejected), and the B/C/D in between are completely ignored.
GRPO assigns gradient signals of varying strengths to each of the 5 candidates - **much more information efficient**.

### GRPO’s new pre-dependency: Reward Model

DPO does not require a reward model (the preference pair itself implicitly defines reward), but GRPO does.

Reward model in industrial practice:
- **OneMall**: Online ranking model (using all user/item/cross features), outputs CTR/CVR/EGPM predictions
- **OneRec**: Dedicated **P-Score model** — multi-tower structure, each tower learns one goal (ctr/lvtr/ltr/vtr), and finally MLP fusion

Our options (wireless sorting model):
- Behavioral signals: `clicked=1, bought=5, exposed_not_clicked=0`
- Offline CTR prediction model
- Embedding similarity (simple but weak)

### Key Findings of GRPO (OneMall)

- **GRPO > DPO**: Better than DPO in all candidate segments (Top10/100/500)
- RL loss weight = 0.5, if it is too large, it will reduce SID accuracy
- Use only 2% training samples for RL
- Calculate normalized advantage for all 768 sampling candidates

## ECPO: Fix training stability of GRPO

**Source**: OneRec (arxiv 2506.13695v4)

### Engineering problem of GRPO: negative advantage gradient explosion

Standard GRPO uses PPO style clips:

```
ratio = π_θ(o|u) / π_old(o|u)
loss = min(ratio * A, clip(ratio, 1-ε, 1+ε) * A)
```

- when A > 0 (good candidate): clip prevents ratio from exceeding 1+ε → **stable**
- When A < 0 (bad candidate): clip only prevents ratio from falling below 1-ε, **but does not prevent ratio from becoming very large** → **gradient explosion**

Intuitive understanding: For a bad candidate, the model may suddenly reduce its probability very low (π_θ → 0) during training,
This causes ratio = π_θ/π_old to become extremely large, multiplied by negative A, resulting in a huge negative gradient.

### Fix for ECPO: Early Clipping

```
π'_old = max( sg(π_θ) / (1+ε+δ),  π_old )

When A < 0, replace π_old with π'_old as the denominator
→ Guarantee ratio never exceeds 1+ε+δ
```

δ = 0.1, which means that the ratio of bad candidates is up to 1+ε+0.1 and will not explode.

In addition, ECPO **removes the KL divergence term**: because OneRec trains RSFT (supervised fine-tuning) and RL simultaneously,
SFT loss itself plays a role in preventing model deviation and does not require additional KL constraints.

### Format Reward: Ensure the legality of the generation

After RL training, the model may generate illegal SIDs (which do not correspond to real items), and the legal rate drops from >95% to <50%.
The reason is **squeezing effect**: when negative advantage compresses the distribution, the probability of legal tokens is reduced to about the same as illegal tokens.

Solution:

```
For randomly sampled K=5 candidates (randomly selected from 128):
Legal SID → advantage = 1 (bonus)
Illegal SID → advantage = 0 (no punishment, just no reward)
```

Why use **random sampling** instead of top-k?
- top-k only makes the top ones legal, while the rest remain illegal - narrow fix
- Random sampling makes the entire distribution tend to be legal, and the final legal rate stabilizes at 95%

The online effect of Format reward: **+0.13% App Stay Time, +0.30% Watch Time**.

### ECPO Key Super Parameters (OneRec)

| Parameter | Value | Description |
|------|-----|------|
| δ (early clip margin) | 0.1 | Controls the maximum ratio of negative advantage |
| Group size (G) | 512 | ~4× inference Pass@K |
| RL user ratio | 1% | Randomly selected from RSFT data flow Medium |
| Format reward K | 5 | Randomly selected from 128 candidate Mediums |
| Sampling strategy | Beam search | Better than top-k/top-p (because SID is a trie structure) |
| Reference model | On-policy | Better than off-policy (offline), but the online difference is small |

### Group Size Ablation (OneRec, Online)

| Group Size | Watch Time | App Stay Time |
|-----------|------------|---------------|
| 0 (no RL) | +4.61% | +1.11% |
| 128 | +5.22% | +1.49% |
| 512 | +5.73% | **+1.82%** |
| 2048 | +5.84% | +1.78% |

The jump from 128→512 is significant, and the marginal decrease from 512→2048. Rule of thumb: group size ≈ 4× inference Pass@K.

## Complete pedigree summary

```
SFT "Push up the probability of correct answers"
│ ✗ I don’t know what mistake the model made
│ ✗ Does not distinguish between "good recommendation" and "accurate prediction"
 ▼
SP-DPO "Pushes correct answers up while pushing down the model's own errors"
│ ✓ Targeting the current confusion area of ​​the model
│ ✗ Only align forecast accuracy, not business goals
│ ✗ You may accidentally damage a good item
 ▼
RF-DPO "Use real user feedback to define good/bad"
│ ✓ Align with business goals (user preferences)
│ ✗ Only looking at pairwise, low information efficiency
│ ✗ No reward model required (use behavioral signals directly)
 ▼
GRPO "Sort the entire set of candidates by reward, group-wise optimization"
│ ✓ All candidates generate gradients and have high information efficiency
│ ✓ Continuous reward signal, not just dichotomy
│ ✗ Require reward model
│ ✗ Negative advantage candidate gradients may explode
 ▼
ECPO "GRPO + early clip to repair stability + format reward to maintain legality"
✓ Stable training
✓ The legality of the generation is guaranteed
✗ The highest engineering complexity (reward model + sampling + clip + format reward)
```

## Practical path to our project

| Phase | Method | Pre-dependency | Complexity | New module |
|------|------|---------|--------|---------|
| 1 | SP-DPO | Only current NTP Model | Low | DPO trainer + preference pair construction |
| 2 | RF-DPO | Requires user behavior tag (clicked/not-clicked) | Low-Medium | Behavior feedback pair construction |
| 3 | GRPO | Requires Training reward model | Medium-High | reward model + group sampling |
| 4 | ECPO | Modify clip + add format reward based on GRPO | High | early clip + format reward + RSFT |

**Current Recommendation**: Start with SP-DPO (minimum front-end dependencies), and gradually upgrade after verifying the DPO framework.
